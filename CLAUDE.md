# debate-judge-bias

oTree 6 experiment: participants arrive from an external Link Survey, join an open chat in one of 82 pre-defined rooms, and discuss until a designated coordinator closes the room. The participant's survey ID is carried in the URL and propagated into the oTree data export so chat behavior can be joined back to survey responses downstream.

## Entry URL contract

Participants reach oTree via:

```
https://<server>/room/debate_room_<N>/?participant_label=<survey_id>
```

- `<N>` — integer 1..82. The survey is responsible for picking which of the 82 rooms this participant goes to (oTree does not decide).
- `<survey_id>` — any string identifying the participant in the upstream survey. oTree stores this automatically as `participant.label`, and we copy it onto `Player.survey_id` so it appears in the per-app CSV export.
- `<survey_id> = "coordinator"` — reserved label. The participant entering with `?participant_label=coordinator` is shown a **Close room** button on the Chat page (see flow below). All other participants ignore this label.

**Public-link entry (no `?participant_label=`).** Participants can also enter via the bare room URL `/room/debate_room_<N>/`. In that case `participant.label` is `None` and `Player.survey_id` falls back to `participant.code` (oTree's always-populated unique code), which is also shown to the participant on the Results page so they can paste it back into the survey manually. Identity across reloads during the session is maintained by oTree's room cookie — so this path assumes one browser per participant. Public-link entrants can never become the coordinator (the label check requires the literal string `coordinator`).

## How the flow works

The study is **single-round, open-attendance, coordinator-closed**. There is no minimum or maximum number of participants, no waiting page, no client-side timer, and no Next button.

1. **Arrival.** Participant hits `/room/debate_room_N/?participant_label=...`. oTree's built-in Rooms feature scopes them to the session bound to that room; the label is stored on `participant.label`. They land directly on the Chat page — no intermediate wait page.
2. **Chat page.** Renders `{{ chat nickname=survey_id channel=<session.code> }}`. All participants of the room's session share the same `chat_channel`, so messages fan out to everyone — including latecomers, who see the full transcript on join.
   - On first render, `vars_for_template` lazily sets `Player.survey_id = participant.label or participant.code`.
   - The page has **no `get_timeout_seconds`** and the default Next button is hidden via CSS. The only way the page advances is via the close signal (next step).
3. **Close signal.** The coordinator (the participant whose `participant.label == "coordinator"`) sees a red **Close room** button. Clicking it calls `liveSend({action: 'close'})`. The `live_method` handler:
   - Verifies the sender is the coordinator (server-side label check — the button is hidden from non-coordinators, but the server re-checks).
   - Sets `session.vars['closed'] = True`.
   - Returns `{0: {closed: true}}`, which oTree broadcasts to **every** player in the group.
   - Because `C.PLAYERS_PER_GROUP = None`, every participant in the session is in the same Group, so the broadcast reaches all open Chat tabs.
4. **Auto-advance.** Each Chat tab's `liveRecv` handler receives `{closed: true}` and submits the page form, advancing to Results.
5. **Latecomers / reloads after close.** On `DOMContentLoaded`, every Chat tab also calls `liveSend({action: 'check'})`. The handler reads `session.vars['closed']` and replies with the current state — so a tab loading after close advances immediately rather than getting stuck in a closed-but-still-rendered chat.
6. **Results.** Static thank-you page. Shows `Player.survey_id` so participants can paste it back into the upstream survey.

`page_sequence = [Chat, Results]`.

## Key constants (`debate/__init__.py`)

- `C.PLAYERS_PER_GROUP = None` — puts every session participant into a single Group, which is what makes `live_method`'s broadcast (`{0: payload}`) reach the whole room.
- `C.NUM_ROUNDS = 1`.
- `COORDINATOR_LABEL = 'coordinator'` — the reserved `participant.label` that grants the Close-room button. Both the template's `is_coordinator` and the server-side `live_method` check against this constant.
- `Player.survey_id: StringField` — mirror of `participant.label` (or `participant.code` fallback) for export.
- `session.vars['closed']: bool` — the close flag. Set by the coordinator's Close click; read by every `liveSend({action: 'check'})` and by the close-broadcast path. One flag per session = one flag per active room instance.

## Operational model: one session per room run

A "room run" maps to one oTree session bound to that room. To run a discussion in `debate_room_5`:

1. Admin → Sessions → New session → pick `debate` config → set `num_participants` to **the maximum expected attendance for this run** (e.g., 100). Bind the session to room `debate_room_5`.
2. Share the room URL with the cohort, including `?participant_label=coordinator` for the moderator and `?participant_label=<survey_id>` for everyone else.
3. Discussion runs open-ended. When the coordinator clicks Close, all tabs advance to Results.
4. To run another discussion in the same room later, create a **new** session — `session.vars['closed']` is per-session, so a fresh session starts unclosed.

`num_participants` is a hard ceiling: oTree won't admit more participants than slots. Set it generously; unused slots cost nothing.

## Data export / linking chat to survey

- Per-app CSV (admin → Data): includes `participant.label` and `player.survey_id`.
- Chat logs: admin sidebar → Chat logs (or `/chat_logs`). Each message row carries the nickname (= survey_id) and the channel (= session code), so `JOIN survey ON survey_id = participant_label` reconstructs chat-plus-survey.

```bash
otree devserver          # http://localhost:8000
otree resetdb --noinput  # wipe & recreate db.sqlite3 — only when adding/renaming
                         # a model field. Removing fields or changing constants
                         # does NOT require this. See "When to resetdb" below.
```

Admin: http://localhost:8000/rooms lists all 82 rooms and their entry URLs.
Test with multiple browser windows hitting the same room URL — one with `?participant_label=coordinator`, the rest with arbitrary labels — and verify the Close button advances every tab.

### When to resetdb (and when NOT to)

`otree resetdb` **wipes all sessions, players, and chat logs** — every past
discussion's transcript and every participant's `survey_id` mapping is gone.
Avoid it whenever the change can be deployed without it.

**Safe to deploy without resetdb (just `git push heroku main && heroku ps:restart`):**
- Removing a model field (column lingers in the DB and is ignored — Django only
  SELECTs declared fields).
- Changing a `C.` constant (e.g. `PLAYERS_PER_GROUP`, `NUM_ROUNDS` from 1→1).
  These are read at session-creation time, so existing sessions keep their old
  group structure and new sessions use the new value.
- Page sequence changes, template edits, new pages, new methods, new
  `live_method` handlers, new `session.vars` keys.
- Anything in `settings.py` (room list, session configs, scaling).

**Requires resetdb (or a manual `ALTER TABLE` via `heroku pg:psql`):**
- Adding a new field to `Player`, `Group`, or `Subsession` — oTree will try to
  SELECT a column that doesn't exist and crash on the first request.
- Renaming a field or changing its type.
- Adding/removing an oTree app from `app_sequence`.

If you must add a field without losing data, the lighter-weight alternative is
to add the column manually:

```bash
heroku pg:psql
ALTER TABLE debate_player ADD COLUMN new_field TEXT NULL;
```

Then deploy. oTree's ORM is happy as long as the column type matches the field
declaration. This preserves existing chat logs and `survey_id` rows.

**The current redesign (open chat + coordinator close) was a removal-only
schema change** (`Group.chat_start_time` dropped, `PLAYERS_PER_GROUP` constant
flipped to `None`, `CHAT_DURATION_SECONDS` removed). It deploys without
resetdb — `git push heroku main && heroku ps:restart` is sufficient. Existing
chat logs are preserved.

## Scaling for high-concurrency sessions (e.g. 256 users)

The default `web=1 worker=1` on Standard-1X is fine for small pilots but tight for
~256 concurrent websockets. The repo is wired so the upgraded fleet can be turned
on for a session and turned off after, without code changes.

**One-time setup (already done in repo):**
- `channels_redis` is in `requirements.txt`.
- `settings.py` switches Django Channels onto Redis when `REDIS_URL` is set; with
  no `REDIS_URL` (local devserver, single-dyno prod), oTree's default layer is used.

**One-time on Heroku (provision Redis; ~$3/mo, leave it on):**

```bash
heroku addons:create heroku-redis:mini      # sets REDIS_URL automatically
git push heroku main                        # deploy the channels_redis change
heroku ps:restart                           # pick up the new CHANNEL_LAYERS
```

**Before a high-load session (≥10 min before participants arrive):**

```bash
heroku ps:type web=standard-2x worker=standard-2x
heroku ps:scale web=2 worker=1
```

**After the session:**

```bash
heroku ps:scale web=1 worker=1
heroku ps:type web=standard-1x worker=standard-1x
# or, if nothing is scheduled soon:
heroku ps:scale web=0 worker=0
```

Heroku bills dynos per-second from the monthly rate, so a 4-hour upgraded window
costs ~$1. The realistic failure mode is forgetting to scale back down — set a
calendar reminder.

## Open decisions (deferred)

- **No-coordinator failure mode.** If the coordinator never joins (or their tab dies before clicking Close), the room never auto-closes. Admin recourse today is to advance participants manually from the admin UI or just let them sit until they leave. Future work could add a hard timeout, an "any admin can close" admin endpoint, or fall back to oTree's session-advance controls.
- **Multiple coordinators.** Any participant entering with `?participant_label=coordinator` becomes a coordinator — there's no enforcement of "exactly one." If two people both join as coordinator, either can press Close. That's fine in practice but worth knowing.
